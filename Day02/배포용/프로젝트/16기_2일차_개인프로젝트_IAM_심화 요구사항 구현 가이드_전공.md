# [I AM] 나만의 소개 공간을 다양한 페이지로 전환하기 — 전공트랙 (진행 가이드)

> 이 문서는 "무엇을 만들지"가 아니라 "어떻게 진행할지"를 다룹니다. 요구사항 목록은 명세서 문서를 참고하세요.

**시작 전에**: 1일차 문서에서 만든 기본형(SPA) 결과물을 다시 열어, 그대로 잘 작동하는지 먼저 확인하세요.

---

## 1. Gemini Canvas 안내

Gemini Canvas는 사용량이 무한하지 않습니다. 사용 중 할당량이 다 되면:
1. Canvas가 아닌 **일반 채팅**으로 전환해서 코드 생성을 요청합니다.
2. VS Code에서 새 html 파일을 생성한 뒤, 생성된 코드를 붙여넣습니다.
3. 이 방식으로 이어서 개발하면 지속적인 작업이 가능합니다.

---

## 2. Git 버전 관리 방법

### 1단계 — 1일차 결과물을 로컬로 가져오기
- Gemini Canvas에서 1일차 결과물(SPA HTML/CSS/JS)을 다운로드(내보내기)
- 작업할 폴더를 만들고 VS Code로 폴더 열기 (예: `startcamp-site/`)
- 다운로드한 파일을 그 폴더 안에 저장

### 2단계 — git 저장소 초기화 및 첫 커밋
VS Code 터미널(또는 Git Bash)에서 아래 명령어를 순서대로 실행합니다.
```
cd startcamp-site
git init
git add .
git commit -m "chore: 1일차 SPA 결과물 초기 커밋"
```
- `git init`: 현재 폴더를 git 저장소로 만듭니다.
- `git add .`: 변경된 파일을 커밋 대상(스테이징)에 올립니다.
- `git commit -m "..."`: 스테이징된 내용을 하나의 기록(커밋)으로 저장합니다.

### 3단계 — 기능 단위로 커밋 나누기
이후 MPA로 구조를 바꾸거나 새 페이지를 추가할 때마다, 한 번에 몰아서 커밋하지 말고 의미 있는 단위로 나눠서 커밋합니다.

**예시**
```
git add .
git commit -m "feat: 네비게이션 바 컴포넌트 확정"

git add .
git commit -m "feat: intro 페이지 추가"

git add .
git commit -m "style: Tailwind CSS로 전환"
```

> 💡 팁: 커밋 메시지 앞에 `feat`(기능 추가), `fix`(버그 수정), `style`(스타일 변경), `chore`(자잘한 작업)처럼 접두어를 붙이면 나중에 기록을 훑어보기 훨씬 편합니다.

- 작업 중간중간 `git status`, `git log --oneline`으로 현재 상태와 기록을 확인하세요.
- 커밋을 쌓아두면, 이후 실수로 코드를 망가뜨려도 이전 커밋으로 되돌아갈 수 있습니다.

## 3. MPA 구성 진행 방법

> 참고 — 핵심 원칙 ("설명"이 아니라 "코드 그대로" 재사용)
> 네비게이션 바를 페이지마다 "넣어줘"라고 말로 설명하면 Gemini가 페이지마다 미묘하게 다른 코드를 생성할 수 있습니다. 그러면 나중에 디자인 하나만 바꿔도 파일마다 결과가 어긋나 있어 손이 더 갑니다.
> 그래서 네비게이션 바를 먼저 코드로 확정한 뒤, 이후 페이지에는 그 코드를 그대로 붙여넣는 방식을 씁니다.

### 1단계 — 네비게이션 바를 먼저 코드로 확정

**요청 예시**
- "아래 페이지들로 연결되는 상단 네비게이션 바 컴포넌트를 HTML+CSS 코드로 만들어줘. 이건 여러 페이지에 똑같이 복사해서 쓸 거야."
- "페이지 목록: {{ 페이지 구성을 작성 }}"
- "현재 보고 있는 페이지는 강조 표시(밑줄 또는 색상 변경)되게 클래스를 넣어줘"
- "모바일에서는 햄버거 메뉴로 접히게 해줘"

→ 이 결과로 나온 코드를 텍스트 에디터에 따로 저장해두세요. 이게 앞으로 계속 재사용할 공통 스펙입니다.

### 2단계 — 페이지별로 Gemini에 요청

각 페이지를 만들 때는 항상 아래 형태로, 1단계 코드를 그대로 붙여서 요청합니다.

- "아래는 확정된 네비게이션 바 코드야. 이 코드를 절대 수정하지 말고 그대로 상단에 삽입해줘."
- "{{ 1단계에서 받은 코드 그대로 붙여넣기 }}"
- "이 페이지는 '{{ 페이지 이름 }}({{ 파일명 }})' 페이지야. 네비게이션 바에서 현재 페이지 표시는 {{ 메뉴 이름 }}에 적용해줘."
- "나머지 본문 내용은 아래 구성으로 만들어줘."
- "{{ 이 페이지에만 들어갈 내용 — 1일차 템플릿의 섹션 구성을 참고해서 채우기 }}"

**사용 예시 (SSAFY 소개 사이트 — 4페이지 분할)**
- "아래는 확정된 네비게이션 바 코드야. 이 코드를 절대 수정하지 말고 그대로 상단에 삽입해줘."
- "{{ 1단계에서 받은 코드 그대로 붙여넣기 }}"
- "이 페이지는 '교육 과정(curriculum.html)' 페이지야. 네비게이션 바에서 현재 페이지 표시는 교육 과정 메뉴에 적용해줘."
- "'기초 → 적용 → 프로젝트'로 이어지는 학습 흐름을 3단계 카드나 타임라인으로 보여주고, 각 단계마다 짧은 설명 문구를 2~3줄씩 넣어줘."

### 3단계 — VS Code에서 조립
- Canvas에서 각 페이지 결과물을 다운로드(내보내기)
- 미리 정한 파일명(예: index.html, intro.html, curriculum.html, support.html...)으로 저장
- Live Server 같은 VS Code 확장으로 로컬에서 링크 이동 테스트
- 이후 미세 조정(공통 CSS 파일로 분리하기 등)은 VS Code에서 직접 진행

> ⚠ 시간 관리 팁: 페이지 수를 2~3개로 제한하는 걸 추천합니다. 페이지가 늘어날수록 1·2단계 반복 횟수가 늘어나서 시간 안에 끝내기 어려워집니다.

## 4. Tailwind CSS 적용 방법

이번에는 프레임워크와 함께 자주 쓰이는 Tailwind CSS로 전환해보는 과제입니다. 클래스 이름만으로 스타일을 입히는 방식이라, 이후 React/Vue 같은 프레임워크로 넘어갈 때도 자연스럽게 이어집니다.

- Tailwind CSS를 CDN 스크립트로 로컬 프로젝트에 추가해보기 (별도 빌드 설정 없이 바로 시작 가능)
- 기존에 직접 작성한 CSS 클래스를 Tailwind 유틸리티 클래스로 하나씩 바꿔보기

**요청 예시**
- "지금 만든 네비게이션 바 코드를 Tailwind CSS 유틸리티 클래스로 다시 작성해줘."
- "기존의 시각적 디자인(색상, 레이아웃, 반응형 동작)은 그대로 유지하면서, 커스텀 CSS 클래스 대신 Tailwind 클래스만 사용해줘."
- "CDN 방식으로 바로 미리보기에서 확인할 수 있게 해줘."

> ⚠ 참고: Tailwind CDN 방식은 빠르게 시작하기엔 좋지만, 실제 서비스에 배포할 때는 빌드 도구(PostCSS 등)를 통한 정식 설치 방식이 권장됩니다. 지금 단계에서는 CDN으로 감을 익히는 것부터 시작하세요.

## 5. 진행 체크리스트

**Git 관리**
- [ ] git init으로 저장소를 초기화했는가
- [ ] 초기 결과물을 첫 커밋으로 남겼는가
- [ ] 기능(페이지/스타일 전환 등) 단위로 커밋을 나누어 기록했는가

**MPA 구성**
- [ ] 네비게이션 바 코드를 먼저 확정하고 별도로 저장해두었는가
- [ ] 모든 페이지에 네비게이션 바 코드를 "그대로" 붙여넣었는가
- [ ] 각 페이지에서 현재 페이지 표시가 올바르게 적용됐는가
- [ ] Live Server로 페이지 간 링크 이동이 정상 작동하는지 확인했는가
- [ ] 페이지마다 폰트/컬러/여백 등 디자인 톤이 어긋나지 않았는가

**Tailwind 전환**
- [ ] Tailwind CDN을 추가하고 기존 디자인이 그대로 유지되는지 확인했는가
- [ ] 기존 CSS가 Tailwind CSS로 정상적으로 전환되었는가

## 6. 용어 설명

- **SPA (Single Page Application)**: 페이지 이동 없이 한 화면 안에서 콘텐츠가 바뀌는 웹사이트 방식.
- **MPA (Multi Page Application)**: 페이지마다 별도의 HTML 파일이 있고, 링크를 클릭하면 새 페이지로 이동(새로고침)하는 전통적인 웹사이트 방식.
- **git**: 파일의 변경 이력을 기록하고 관리하는 버전 관리 시스템. 언제, 무엇이, 왜 바뀌었는지 추적하고 이전 상태로 되돌릴 수 있습니다.
- **커밋 (commit)**: 특정 시점의 변경 내용을 저장소에 하나의 기록으로 저장하는 단위.
- **저장소 (repository)**: git으로 관리되는 프로젝트 폴더. `git init`으로 만듭니다.
- **게이미피케이션 (Gamification)**: 포인트, 진행률, 배지처럼 게임적 요소를 넣어 재미와 몰입감을 주는 기법.
- **Tailwind CSS**: 미리 정의된 클래스 이름(예: `text-center`, `bg-blue-500`)을 조합해서 스타일을 입히는 CSS 프레임워크. 커스텀 CSS 파일을 따로 작성하지 않아도 됩니다.
- **컴포넌트 (Component)**: 네비게이션 바, 버튼처럼 화면의 일부를 재사용 가능한 조각 단위로 분리한 것. React, Vue 같은 프레임워크에서 자주 쓰는 개념입니다.
